# Phase A: 실현 가능성 스파이크

**목적**: QR 오버래핑 접근이 표준·물리·운영 제약 안에서 성립하는지 조기 검증한다.

**선행 조건**: Phase B 본격 구현(모듈 단위 테스트 확장) 전에 이 단계를 통과하거나, 실패 시 오버래핑 정의를 수정한다.

## A-1. 스택 성립성

| ID | 설명 | 기존 매핑 |
|----|------|-----------|
| A-ASM1 | 선택 라이브러리 조합으로 QR 생성·판독 기본 동작 | T-ASM1 |
| A-ASM2 | 단일 payload 기준 최대 수용량, ECC별 용량, 버전 한계 | T-ASM2 |
| A-ASM3 | 생성기 출력이 독립 판독 경로(다른 라이브러리/OS 카메라 앱 등)에서도 읽히는지 | 신규 |

## A-2. 오버래핑 가설 검증

상세 시나리오는 [overlap-and-independent-readpath.md](overlap-and-independent-readpath.md)다.

| ID | 설명 |
|----|------|
| A-HYP1 | 단일 QR에 A/B/C를 구분 가능한 표현으로 실을 수 있는가 |
| A-HYP2 | 일반 스캔 경로는 공개 레이어만, 확장 해석 경로에서만 추가 레이어 복원 |
| A-HYP3 | baseline 단일 payload QR 대비 판독 가능성 저하가 허용 범위인가 |
| A-HYP4 | 시각적 동일성 또는 단일 심볼 유지 조건 정의 및 검증 |

## A-3. 환경 내구성 스파이크

| ID | 설명 | 기존 매핑 |
|----|------|-----------|
| A-ENV1 | 화면 표시·조도 변화별 판독 | (EXP 일부) |
| A-ENV2 | 회전·기울기·거리 | 신규 |
| A-ENV3 | 리사이즈·JPEG·부분 왜곡·일부 가림, 레이어별 영향 | T-EDGE3, T-EDGE4 |
| A-ENV4 | payload 크기 증가에 따른 성공률·속도 곡선 | T-EDGE1 연관 |

## A-4. 측정 프로토콜 고정

측정 정의는 [metrics-protocol.md](metrics-protocol.md)에 단일 원본으로 기록한다. Phase A에서는 아래를 **문서 승인** 수준까지 고정한다.

| ID | 내용 |
|----|------|
| A-MET1 | 스캔 성공률: 디바이스·조명 매트릭스, 반복, 실패 판정 |
| A-MET2 | 레이턴시: 지표(평균 vs 분위수), I/O 포함 여부, cold/warm |
| A-MET3 | 무결성: 변조 유형·표본 수·탐지율 계산 |
| A-MET4 | 라우팅: 100케이스 생성 규칙·safe default |

## Phase A 종료 조건

- 오버래핑 방식이 최소 1개 이상 현실적 후보로 남는다.
- `metrics-protocol.md`가 Phase B 테스트의 pass/fail 기준으로 사용 가능하다.
- 실패 시: 구조 단순화 또는 오버래핑 정의 수정 후 Phase B로 넘어가지 않는다.
