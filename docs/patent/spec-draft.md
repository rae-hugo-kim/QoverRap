# QoverwRap 특허 명세서 초안

> **상태**: `ai_generated` — AI 작성 본인 출원 (KEAPS) 워크플로우. 변리사 검토 단계 없음. 14-round adversarial review 완료, KEAPS 형식 변환 후 출원.
> **검증 정책**: 청구항 후보의 신규성/진보성은 `docs/research/prior-art-survey.md` (KIPRIS + EPO OPS + Google Patents + 비특허 문헌, **출원 직전 12개월 재조사 §10 포함**) 결과에 근거하며, "QR + 서명 일반"은 신규성의 핵심 근거로 쓰지 않는다. 핵심 차별점은 **Layer A 평문 prefix + delimiter-framed trailer + versioned binary header + safe-fallback resolver의 구체적 결합**이다.
> **관련 문서**: 코어 구현 `src/qoverwrap/`, 통합 테스트 `tests/test_*.py`, 데모 앱 `demo/`, 선행기술 조사 `docs/research/prior-art-survey.md`.

## 1. 발명의 명칭

**표준 QR 리더 호환 공개 계층과 내장 서명 트레일러를 포함하는 단일 QR 페이로드 검증 시스템과 그 방법** (가제)

영문: *System and Method for Offline Verification of a Layered Single QR Payload with a Plaintext Public Prefix and Embedded Signature Trailer*

---

## 2. 기술 분야

본 발명은 QR 코드(Quick Response Code, ISO/IEC 18004) 기반 데이터 인코딩 및 검증 분야에 관한 것으로, 보다 상세하게는 **단일 QR 코드의 표준 데이터 필드 내에서 복수의 논리 계층(layer)을 구조화하여 직렬화하고**, 디지털 서명을 동일 페이로드 내에 내장하며, 디코더 측에서 검증 결과 및 출력 정책에 따라 응용 프로그램에 반환되는 계층 데이터의 범위를 결정하는 시스템 및 그 방법에 관한 것이다.

---

## 3. 발명의 배경이 되는 기술

### 3.1 표준 QR 코드의 한계

범용 QR 리더는 통상 QR 심벌에 인코딩된 데이터를 하나의 디코딩 결과 문자열로 반환하며, 해당 문자열 내부의 정보에 대하여 권한별 논리 계층, 서명 검증 상태 또는 출력 정책을 표준적으로 제공하지 않는다. 디코더 측 출력 정책 분기, 위변조 탐지, 발급자 진위 확인을 단일 QR 페이로드 내부 정보만으로 수행하기 어렵고, 일반적으로 외부 검증 서버 또는 별도의 인증 인프라에 의존한다.

### 3.2 종래 기술의 분류 및 한계 (선행기술 조사 결과)

선행기술은 다음 5개 군으로 분류된다 (`docs/research/prior-art-survey.md` §3, §5, §7 참조). 본 발명의 차별점은 어느 단일 군의 일반 개념에도 속하지 않으며, 특히 (마) 군과는 후술하는 구체적 wire-format 결합으로 구별된다.

### 3.3 EPO OPS 보강 검색 결과의 해석

다음 키워드 조합은 EPO OPS 검색에서 직접 hit가 없었다 (`prior-art-survey.md` §6-5).

- `cpc=G06K19/06 AND ti,ab=(signature AND QR)` → 0 hit
- `ti,ab=(QR AND ed25519)` → 0 hit
- `ti,ab=(QR AND (delimiter OR "payload structur*"))` → 0 hit

다만 본 발명은 위 0-hit 결과 자체를 신규성의 핵심 근거로 사용하지 않는다. QR 또는 barcode 페이로드에 디지털 서명을 포함하여 인증을 수행하는 선행기술(§3.2 (마) 군)은 별도로 존재하므로, 본 발명의 차별점은 "QR 내 서명" 일반 개념이 아니라, **(i) Layer A를 표준 QR 리더 판독 결과의 선두 평문으로 유지하고, (ii) 미리 정해진 구획자 이후의 트레일러에 버전·길이 정보를 포함하는 이진 헤더와 후속 계층 데이터를 텍스트 안전 인코딩으로 결합하며, (iii) 파싱 또는 서명 검증 실패 시 Layer A만 반환하는 safe-fallback resolver 정책을 적용**하는 구체적 결합에 둔다.

## 4. 발명이 해결하려는 과제

1. 단일 표준 QR 코드의 데이터 필드 문자열 내에 **복수의 논리 계층**을 QR 모듈 또는 ECC 코드워드의 은닉 채널 변조 없이 구조화 저장한다.
2. 검증 대상 데이터와 서명값을 동일 페이로드 내부에 포함시켜, **온라인 검증 서버 또는 블록체인 조회 호출 없이** 사전 보유 또는 외부 입력으로 제공된 공개키만으로 위변조 검증이 가능하게 한다.
3. 디코더 또는 응용 프로그램이 사용자 또는 후속 처리 모듈에 반환하는 계층 데이터의 범위를 **출력 정책에 따라 결정**하고, 실패 유형을 두 그룹으로 구분하여 강등하는 안전 강등(safe-fallback) 정책을 보장한다. 즉 구획자 최초 출현 위치 기준 Layer A·트레일러 분리 이후의 트레일러 디코딩·이진 헤더 검증·헤더 길이 필드와 실제 프레임 길이 간의 정합성 검증 실패의 구조 파싱 실패 그룹에서는 제2 출력 수준 및 제3 출력 수준 양자 모두에서 Layer A 만을 반환하고, 공개키 부재·서명 검증 실패의 검증 실패 그룹에서는 서명 검증이 요구되는 제3 출력 수준에 한하여 Layer A 만을 반환한다.
4. **표준 QR 리더와 부분 호환**되어, 본 발명을 모르는 디코더도 Layer A를 선두 평문으로 정상 판독하며, 후속 트레일러는 표준 리더의 동작을 방해하지 않는다.

## 5. 과제 해결 수단 (요약)

본 발명은 다음 구성을 결합한다.

(a) **페이로드-레벨 구조화**: QR 코드의 표준 데이터 필드 *문자열 내용 안에서* 미리 정해진 구획자(delimiter) 와 이진 헤더(binary header) 를 사용하여 공개 계층 Layer A(평문 텍스트)와, 후속 계층 데이터(예: 컨텍스트 메타데이터 계층 Layer B 및 디지털 서명 계층 Layer C)를 직렬화한다. 후속 계층은 텍스트 안전 인코딩(예: base64)으로 트레일러에 결합된다.

(b) **내장 서명**: 동일 페이로드 내부의 정규화된 서명 대상 메시지(§8.1.1, 예: `magic("QWR1")`||version||len(A)||Layer A UTF-8||len(B)||Layer B)에 대한 디지털 서명(예: Ed25519)을 Layer C 로 동일 QR 페이로드 안에 내장한다. 검증기는 사전에 보유하거나 외부에서 제공된 공개키를 사용하여 온라인 검증 서버 호출 없이 서명 검증을 수행한다.

(c) **출력 정책 기반 리졸버**: 디코더 또는 응용 프로그램에 반환되는 계층 데이터의 범위를 출력 정책 매개변수(이하 "접근 수준")에 따라 결정한다. 본 발명에서 접근 수준은 페이로드 자체의 암호학적 판독 제한을 의미하지 않으며, 응용 프로그램이 사용자 또는 후속 처리 모듈에 반환하는 계층 데이터의 범위를 결정하기 위한 출력 정책 매개변수를 의미한다. 강등은 실패 유형에 따라 두 그룹으로 구분되며, 구조 파싱 실패 그룹(구획자 최초 출현 위치 기준 Layer A·트레일러 분리 이후의 트레일러 디코딩·이진 헤더 검증·길이 정합성 검증 실패)에서는 제2 및 제3 출력 수준 양자 모두 Layer A 만 반환하고, 검증 실패 그룹(공개키 부재·서명 검증 실패)에서는 제3 출력 수준에 한하여 Layer A 만 반환한다 (safe-fallback).

(d) **표준 리더 부분 호환**: Layer A 가 평문으로 페이로드 선두에 위치하므로 표준 QR 리더는 Layer A를 의미 있는 텍스트로 그대로 표시하며, 후속 트레일러는 표준 리더 동작을 방해하지 않는다.

## 8. 발명을 실시하기 위한 구체적인 내용

### 8.1 페이로드 비트 레이아웃 (Fig. 1)

```
[단일 페이로드 문자열 (10)]

┌─────────────────────────────────────────────────────────────────┐
│ Layer A (11) — UTF-8 평문 문자열                                  │
│   예: "qwr:tigers-2026|VIP seat A1"                              │
├─────────────────────────────────────────────────────────────────┤
│ 구획자 (12) DELIMITER = "\n---QWR---\n"                          │
│   (UTF-8/ASCII 기준 고정 11 바이트)                               │
├─────────────────────────────────────────────────────────────────┤
│ 트레일러 (13) = base64( 이진 헤더(14) ‖ Layer B(15) ‖ Layer C(16) )│
│                                                                 │
│ 이진 헤더 (14) — 5 바이트, big-endian:                            │
│   ├─ version 필드 (14a) : uint8   (현재 0x01)                    │
│   ├─ b_len 필드   (14b) : uint16  (Layer B 바이트 길이)            │
│   └─ c_len 필드   (14c) : uint16  (Layer C 바이트 길이)            │
│                                                                 │
│ Layer B (15) — 컨텍스트 메타데이터 계층, 임의 바이너리 (≤ 65535 B)  │
│ Layer C (16) — 디지털 서명 계층 (예: Ed25519 = 64 바이트)          │
└─────────────────────────────────────────────────────────────────┘
```

[규칙]
- Layer B 가 입력되지 아니하여 후속 계층이 부재인 경우 DELIMITER 와 트레일러를 생성하지 아니하고 Layer A 만으로 페이로드를 구성한다 (단순 모드). Layer B 가 입력되는 경우 — Layer B 가 영(零) 바이트 길이로 입력되는 경우를 포함 — 인코더는 Layer A 및 Layer B 에 기초한 서명 대상 메시지에 대하여 비밀키로 서명함으로써 Layer C 를 생성한 후 DELIMITER 와 트레일러를 생성한다 (복합 모드). Layer C 단독의 입력 또는 Layer B 와 Layer C 가 비대칭으로 결합되는 형태는 청1 의 범위에 속하지 아니한다.
- Layer A 는 DELIMITER 문자열을 포함할 수 없다 (인코더가 거부).
- 표준 QR 리더는 Layer A + DELIMITER + (텍스트 안전 인코딩 트레일러) 전체를 단일 평문으로 인식한다 → 본 발명을 모르는 디코더에서는 Layer A 가 의미 있는 정보로, 트레일러는 무관한 문자열로 보인다.

### 8.1.1 서명 대상의 정규화 (canonicalization)

서명 검증 결과의 구현 간 일관성을 보장하고, 서명 대상에 포함되는 버전 및 길이 관련 필드의 변조에 대한 무결성을 함께 확보하기 위하여, 일 실시예에서 서명 대상 메시지(signing message)는 다음과 같은 정규화된 바이트열로 구성된다.

```
signing_message =
    magic("QWR1") (4 바이트)
 || uint8(version)
 || uint16_be(len(layer_a_utf8))
 || layer_a_utf8
 || uint16_be(len(layer_b))
 || layer_b
```

또한 복합 모드 (구획자 및 트레일러가 존재하는 모드) 에서 이진 헤더의 Layer B 길이 필드가 영(零) 으로 설정된 페이로드는, 본 정규화 규칙에 따라 Layer B 의 길이 필드를 0 으로, Layer B 의 바이트열을 영 바이트 길이로 결정론적으로 직렬화하여 서명 및 검증이 가능하다. 단순 모드 (구획자 및 트레일러가 부재한 모드) 에서는 이진 헤더 자체가 페이로드 wire format 에 존재하지 아니하므로, Layer B 가 영 바이트 길이로 존재하는 복합 모드의 상태와 Layer B 자체가 페이로드에 포함되지 아니하는 단순 모드의 상태가 wire format 차원에서 결정론적으로 구별된다.

다른 실시예에서는 위 매직 식별자를 생략하고 헤더 바이트열을 그대로 서명 대상에 포함시키거나, Layer B 가 비어있는 경우 길이 필드만 0 으로 설정하는 등의 변형이 가능하다. 또한 본 발명의 후속 확장으로서 알고리즘 식별자(alg_id), 텍스트 안전 인코딩 식별자(encoding_id), 플래그(flags), 키 식별자(key_id) 등의 추가 메타데이터 필드가 이진 헤더에 도입되는 경우, 해당 필드들이 서명 대상 메시지에 포함되도록 직렬화 규칙을 확장할 수 있다. 어느 변형에서도 서명 대상 메시지는 (i) 버전, (ii) Layer A의 UTF-8 바이트열, (iii) Layer B의 바이트열 및 그 길이 정보를 포함하며, 추가 헤더 필드가 존재하는 실시예에서는 그 필드들도 함께 포함되도록 결정론적으로 직렬화된다.

### 8.2 인코더 동작 (Fig. 2)

```
[S100] 입력 인터페이스 (110)
       layer_a (Layer A, 11), layer_b (Layer B, 15) [optional],
       signing_key (서명 키 저장부 120 로부터)
 │
 ▼
[S101] 입력 검증
       layer_a 에 구획자(12) 미포함, layer_b 가 제공된 경우 len(layer_b) ≤ 65535
 │
 ▼
[S102] 단순/복합 모드 분기 ─── layer_b 가 입력되지 아니하였는가?
 │              │
 │              └── Yes (양자 부재 모드) ──→ [S106] 단순 모드 출력
 │                          return layer_a (= 단일 페이로드 문자열, 10)
 │              No (양자 포함 모드 — len(layer_b) = 0 인 경우 포함)
 ▼
[S102.5] 정규화된 서명 대상 메시지 생성 및 Layer C(16) 생성
       signing_message = canonical_signing_message(layer_a, layer_b)
       layer_c = sign(signing_key, signing_message)
       (Layer C 는 인코더가 생성하는 출력값이며 입력값이 아님)
 │
 ▼
[S103] 이진 헤더 (14) 생성
       header = pack(">BHH", 0x01, len(layer_b), len(layer_c))
       (version 14a, b_len 14b, c_len 14c)
 │
 ▼
[S104] 프레임 결합 + 텍스트 안전 인코딩
       frame   = header || layer_b || layer_c
       trailer = base64.b64encode(frame).decode()  → 트레일러 (13)
 │
 ▼
[S105] 복합 모드 출력
       return layer_a + DELIMITER(12) + trailer(13)
       (= 단일 페이로드 문자열, 10)
```

### 8.3 디코더 + 리졸버 (Fig. 3)

```
[S200] 페이로드 입력 (210)
       payload (단일 페이로드 문자열, 10)
       access_level ∈ {제1, 제2, 제3 출력 수준}
       public_key (선택, 230 으로부터 또는 외부 입력)

── 디코더 (220) ──────────────────────────────────────────────────

[S201] 구획자 (12) 존재 검사
       payload 에 DELIMITER 미포함?
        │              │
        │              └── Yes ──→ return (payload as Layer A(11), absent, absent)
        │                          (Layer B / Layer C 는 페이로드에 부재(absent) 또는
        │                           미설정(unset) 상태로 표현하며, 복합 모드의
        │                           영 바이트 Layer B 와 출력 데이터 구조에서 구별)
        No
        ▼
[S202] split(DELIMITER, max=1) → Layer A(11), 트레일러(13)
        ▼
[S203] 트레일러 디코딩 + 헤더 파싱 (v1: strict)
       frame = base64.b64decode(trailer, validate=True)
       (version, b_len, c_len) = unpack(">BHH", frame[:5])  ← 이진 헤더 (14)
       len(frame) == 5 + b_len + c_len 이어야 함 (trailing junk 거부)
       추출 → Layer B (15) = frame[5 : 5+b_len]
       추출 → Layer C (16) = frame[5+b_len : 5+b_len+c_len]

── 리졸버 (250) ──────────────────────────────────────────────────

[S204] 출력 수준 정규화
       (미지 access_level → 제1 출력 수준 = public)
        ▼
[S205] decode_layers 시도 → 실패 시 [S207] safe-fallback
        ▼
[S206] 출력 수준 분기
        ├─ 제1 (public)        → return (Layer A, None, vr=public)
        ├─ 제2 (authenticated) → return (Layer A, Layer B, vr=unverified)
        └─ 제3 (verified)
                │
                ├─ public_key 부재 → [S207] safe-fallback
                │
                ├─ 검증부 (240): verify_signature(pk, Layer A, Layer B, Layer C)
                │      │
                │      ├─ valid   → return (Layer A, Layer B,
                │      │                    vr=valid,
                │      │                    signature=optional (diagnostic))
                │      │
                │      └─ invalid → [S207] safe-fallback
                │
[S207] safe-fallback 강등
       Layer B, Layer C, Layer B 기반 렌더링 결과 반환 차단
       Layer A 및 실패 상태 정보만 포함하는 공개 수준 결과로 강등
        ▼
[S208] 반환 출력 인터페이스 (260) → 사용자 또는 후속 응용 모듈
```

[안전 기본값] 본 발명의 강등 동작은 실패 유형에 따라 두 그룹으로 구분된다. (a) 구조 파싱 실패 그룹 — 구획자 최초 출현 위치 기준 Layer A 와 트레일러가 분리된 이후에 발생하는 트레일러 디코딩 실패, 이진 헤더의 버전 또는 길이 검증 실패, 또는 이진 헤더의 길이 필드와 디코딩된 프레임의 실제 길이 간의 정합성 검증 실패 — 의 경우, 제2 출력 수준 또는 제3 출력 수준이 요청된 경우에도 Layer A 만 반환되도록 강등된다. (b) 검증 실패 그룹 — 공개키 부재, 서명 검증 실패 — 의 경우, 제3 출력 수준이 요청된 때에 한하여 Layer A 만 반환되도록 강등된다. 제2 출력 수준은 서명 검증을 요구하지 아니하므로 검증 실패 그룹은 제2 출력 수준에서 강등 사유에 해당하지 아니하며, Layer B 가 미검증 상태로 표시되어 반환된다. 강등이 발동된 어떠한 실패 경로에서도 결과는 공개 계층(Layer A) 을 초과하는 정보를 반환하지 않는다. 본 강등 정책은 단순한 통상적 오류 처리가 아니라, 본 발명의 계층형 페이로드 구조 및 트레일러 검증 절차와 결합된 반환 데이터 제한 정책이다.

[접근 수준의 정의] 본 명세서에서 "접근 수준(access level)"은 디코더 또는 응용 프로그램이 사용자 또는 후속 처리 모듈에 반환하는 계층 데이터의 범위를 결정하기 위한 출력 정책 매개변수를 의미하며, 페이로드 자체의 암호학적 판독 제한을 의미하지 않는다. 일 실시예에서 출력 수준은 제1 출력 수준(공개 / public), 제2 출력 수준(구조화된 메타데이터 반환 / authenticated, 단 Layer B 의 암호학적 무결성이 검증되지 않았음을 표시), 제3 출력 수준(서명 검증 성공 시에 한하여 검증된 데이터로 반환 / verified)의 3단계로 구성된다. 제2 출력 수준에서 반환되는 Layer B 는 검증된 데이터가 아니며, Layer B 가 검증된 데이터로 반환되는 것은 제3 출력 수준에서 서명 검증이 성공한 경우에 한한다.

### 8.4 응용 레이어 — 발급자 라우팅 (Fig. 4)

본 절은 **본 발명의 선택적 실시예** 로서, Layer A 의 선두 영역에 포함된 발급자 식별자를 이용하여 검증용 공개키를 선택하는 응용 레이어의 라우팅 구성을 설명한다.

[Layer A (11) 발급자 ID 컨벤션]
```
qwr:<issuer-id>|<human-readable message>
```

[신뢰 레지스트리 (230) — 운영사 측]
```python
TRUST_REGISTRY = {
    "tigers-2026":      ed25519_pub_A,
    "violet-fandom":    ed25519_pub_B,
    "comic-con-2026":   ed25519_pub_C,
    ...
}
```

[검증 시 라우팅 흐름]
```
[S300] QR 스캔 (210)
        ▼
[S301] Layer A (11) 추출
        ▼
[S302] "qwr:" 프리픽스 파싱 → issuer-id 분리
        ▼
[S303] TRUST_REGISTRY (230)[issuer-id] → public_key 조회
        ▼
[S304] 검증부 (240) 의 verify_signature() 호출
        ▼
[S305] 결과를 응용 UI(시각 테마/카드 렌더링)에 반영
```

이 구성에서 시각 테마(색·로고)는 issuer-id 와 1:1 대응되어, **사용자가 보는 시각적 표식이 곧 내부적으로 사용되는 검증 키의 단서**가 된다. 즉, 응용 측에서 마케팅 요소와 기술적 검증 루트가 분리되지 않는다.

### 8.5 데모 앱 실시예 (본문 설명 — 본 출원에서는 화면 캡처 도면을 첨부하지 아니함)

본 발명의 실시를 위한 일례로서, `demo/` 디렉토리에 React + FastAPI 기반 시연 앱이 구현되어 있다. 본 절은 해당 앱의 동작을 본문으로 기재하며, 화면 캡처 자체는 본 출원의 도면으로 첨부하지 아니한다 (도면 첨부 시의 형식 결함 가능성으로 인함). 본 실시예의 청구항 뒷받침은 §8.1 내지 §8.4 및 §8.6 의 시스템·플로우 도면, 그리고 본 §8.5 의 본문 설명으로 충분히 이루어진다.

#### 8.5.1 Bare 모드 (청구항 본체 시각화)

3-column 비교 화면에서 동일 QR 페이로드가 출력 수준별로 다음과 같이 사용자/응용 프로그램에 반환된다. 표준 QR 리더는 페이로드 문자열 전체(Layer A + 구획자 + 트레일러)를 판독하지만, 본 도면의 Public 컬럼은 *판독 결과 중 사람이 읽을 수 있는 선두 평문(Layer A prefix)*을 발췌 표시한 것이다. 즉, 후속 트레일러는 표준 리더에 의해 판독되되, 본 발명을 모르는 응용 프로그램에서는 그 구조적 의미가 해석되지 않을 뿐이다.

- Public 컬럼 (Layer A prefix view): 표준 QR 리더 판독 결과 중 의미 있는 선두 평문 (Layer A) 만을 발췌 표시. 후속 트레일러는 본 컬럼의 응용 정책상 해석·표시하지 않음.
- Authenticated 컬럼 (제2 출력 수준 / 미검증 구조화 메타데이터): Layer A 및 Layer B 를 반환하되, Layer B 의 검증 상태를 미검증(UNVERIFIED) 으로 표시.
- Verified 컬럼 (제3 출력 수준 / 검증된 메타데이터): 서명 검증이 성공한 경우에 한하여 Layer A 및 Layer B 를 검증된 데이터로 표시하고, Layer C 는 서명 검증에 사용된 검증 재료로서 선택적으로 축약 표시하며, 검증 결과 배지(VALID / INVALID)를 함께 표시.

#### 8.5.2 Themed 모드 (응용 예시)

동일한 QoverwRap wire format 및 resolver 절차를 사용하는 서로 다른 발급자별 페이로드를, 각 발급자 테마로 렌더링했을 때의 사용자 화면 (본 출원 도면 미첨부).

- (Tigers): 야구 입장권 카드 — 좌석/구역/게이트/날짜/상대팀
- (Violet): 팬덤 멤버십 카드 — Tier/Member ID/독점 콘텐츠
- (Comic Con): 출입 배지 — 패스 등급/Hall/Panel

세 카드는 동일한 페이로드 구조와 검증 절차를 사용하되, Layer A/B 의 내용 및 응용 레이어의 시각 테마가 발급자별로 다르다.

#### 8.5.3 위변조 검출

서명 후 Layer B 가 변경된 경우, Verified 컬럼 카드는 §8.3 의 안전 강등 정책에 따라 Layer B 의 검증된 데이터 반환을 차단하고 Layer A 만을 반환하는 공개 수준으로 강등되며, INVALID 배지가 표시된다. 동일 페이로드를 원본 데이터로 복원하면 즉시 검증이 성공하여 VALID 로 회복된다.

[구현 비종속성]

본 시스템 구성은 단일 컴퓨팅 디바이스 상의 모듈 분할로 실시될 수도 있고, 발급 측·검증 측이 물리적으로 분리된 분산 환경에서 실시될 수도 있다. 신뢰 레지스트리는 메모리 내 자료구조, 파일, 데이터베이스, 또는 외부 입력 채널 중 어느 것으로도 구현될 수 있으며, 본 발명의 청구범위는 특정 저장 매체나 구현 형태에 한정되지 않는다.

[청구항 11 변형 후보 — 출원 직전 정제 시 검토]

청구항 11 의 시스템항은 출원 직전 정제 단계에서 보다 전통적인 컴퓨터 시스템 청구항 형태로 변환될 수 있다. 일례로 다음 형태가 가능하다.

> 하나 이상의 프로세서; 및 상기 하나 이상의 프로세서에 의해 실행될 때 청구항 1, 청구항 2, 청구항 7 또는 청구항 8 의 단계를 수행하게 하는 명령어를 저장하는 메모리; 를 포함하는 단일 QR 페이로드 계층형 데이터 처리 시스템.

본 변형은 본 명세서가 제시하는 인코더·디코더·검증부·리졸버 모듈의 기능적 구성과 동등하며, 본 발명의 청구범위에 포함되는 실시예이다.

```
       반환 데이터를 단말 사용자에게 전달
```

여기서 휴대용 단말은 스마트폰, 태블릿, 웨어러블 디바이스 등 이미지 센서·표시부·통신 모듈을 갖춘 임의의 단말로 실시될 수 있으며, 표시부는 디스플레이를 포함하는 시각 출력 수단을, 음성 출력부는 스피커를 포함하는 청각 출력 수단을, 햅틱 출력부는 진동 또는 변형을 포함하는 촉각 출력 수단을 각각 의미한다. 본 발명의 청구항 2 의 디코더·검증부 동작 자체는 휴대용 단말의 내부 프로세서에서 수행될 수도 있고, 휴대용 단말이 페이로드 문자열을 본 발명의 시스템서버에 전송한 후 결과만을 수신하는 분산 구성으로 실시될 수도 있다. 어느 실시예에서도 청구항 2 본문의 "본 발명의 검증기 및 본 발명의 시스템서버 외부의 제3자 검증 인프라에 대한 호출은 수행하지 아니한다" 한정은 검증 절차에 적용되는 한정이며, 사용자 단말 ↔ 본 발명의 시스템서버 간의 페이로드 전달은 본 한정의 외부 호출에 해당하지 아니한다 (청구항 2 단서 조항과 동일한 취지).

[8.7.2 검증된 빈 Layer B 의 보존 정책 — 청구항 13]

본 발명의 일 실시예에서, 발급자는 컨텍스트 메타데이터가 존재하지 않는 페이로드에 대해서도 디지털 서명을 부착할 수 있다. 즉, Layer A 만 서명 대상으로 하되, 후속 계층 데이터 중 Layer B 의 길이를 0으로 설정하여 인코딩하는 실시예가 존재한다 (§8.1.1 의 정규화에서 `uint16_be(len(layer_b)) = 0x0000`, Layer B 바이트열 미존재).

이러한 페이로드를 디코딩하여 청구항 7 의 제3 출력 수준으로 처리할 때, 본 발명은 다음의 보존 규칙을 적용한다.

```
verify_signature(pk, Layer A, Layer B = b"", Layer C) == valid
            ▼
        리졸버 출력:
            Layer A   : <검증된 평문>
            Layer B   : b""                    ← 빈 바이트열을 그대로 보존
            verified  : True
        ※ Layer B 를 None, null, undefined, "" → falsy 변환 후 제거,
          또는 키 자체의 부재 (missing key) 로 직렬화하지 아니함.
```

본 보존 규칙은 다음 두 가지 의미를 가진다.

(i) **검증된 상태와 미존재 상태의 분리** — Layer B 가 검증되어 빈 값으로 확정된 경우와, Layer B 가 페이로드 자체에 부재하여 인코딩되지 않은 경우(청구항 1(f) 의 단순 모드)는 출력 수준 결과에서 구별되어야 한다. 본 발명은 제3 출력 수준에서 전자를 `Layer B = b""` (검증됨, 길이 0) 으로, 후자를 미설정(unset) 또는 단순 모드 응답으로 구별 표현한다.

(ii) **응용 프로그램 측 직렬화 규약** — 본 보존 규칙은 디코더·리졸버의 내부 자료구조뿐 아니라, 응용 프로그램이 후속 모듈(예: HTTP API, gRPC, UI 컴포넌트) 로 데이터를 직렬화하는 경계에서도 유지된다. 일 실시예에서, 본 발명의 시스템은 응답 직렬화 시 빈 바이트열을 "(empty — verified)" 와 같은 명시적 시각 또는 의미적 표시로 출력하여, 빈 바이트열을 부재(absent) 와 구별하도록 한다.

본 보존 규칙이 적용되지 않는 종래의 trustness-falsy 직렬화(`if x else None`) 에서는, 검증된 빈 Layer B 가 wire 상에서 부재로 강등되어 청구항 7(iii) 의 "검증된 데이터" 의미가 손실된다. 본 발명은 디코더·리졸버 내부 invariant 및 응용 직렬화 경계 양쪽에서 본 보존 규칙을 강제함으로써 해당 의미 손실을 차단한다.

[8.7.3 시스템 결합 실시예 — 청구항 15]

청구항 15 는 §8.6 Fig. 8 의 시스템 구성에서 검증부 (240) 가 §8.1.1 의 정규화된 바이트열을 재구성하여 서명 검증을 수행하고, 리졸버 (250) 가 §8.7.2 의 빈 바이트열 보존 정책을 적용하도록 결합된 실시예에 해당한다. 본 결합 실시예에서, 휴대용 단말 환경(청구항 14) 또는 §8.6 의 분산 환경 어느 쪽에서도 청구항 12·13 의 보호 대상이 동일하게 보존된다.

### 9.1 독립항 후보

**[청구항 1] 계층형 단일 QR 페이로드 생성 방법**

QR 코드의 표준 심벌 체계에 의해 인코딩 가능한 단일 페이로드 문자열을 생성하는 방법으로서,
(a) 표준 QR 코드 데이터로 인코딩되었을 때 표준 QR 리더의 판독 결과 문자열의 선두에 평문으로 위치하는 공개 텍스트 계층인 Layer A 를 입력받는 단계;
(b) Layer A 와 구별되는 컨텍스트 메타데이터 계층인 Layer B 를 입력받거나 입력받지 아니하는 단계 — Layer B 를 입력받는 경우 양자 포함 모드에 해당하고, 입력받지 아니하는 경우 양자 부재 모드에 해당하며, Layer B 는 영(零) 바이트 길이로서 입력되는 경우를 포함하고, 디지털 서명 계층인 Layer C 단독을 입력받는 형태는 본 청구항의 범위에서 제외됨;
(c) 상기 양자 포함 모드에서, Layer A 및 Layer B 에 기초하여 결정론적으로 직렬화된 서명 대상 메시지에 대하여 사전 보유된 비밀키로 서명함으로써 디지털 서명 계층인 Layer C 를 생성하는 단계;
(d) 상기 양자 포함 모드에서, 버전 정보, Layer B 의 바이트 길이를 나타내는 길이 필드, 및 Layer C 의 바이트 길이를 나타내는 길이 필드를 포함하는 이진 헤더를 생성하는 단계;
(e) 상기 양자 포함 모드에서, 상기 이진 헤더, Layer B, 및 Layer C 를 순차적으로 결합한 프레임을 텍스트 안전 인코딩 방식으로 인코딩하여 트레일러를 생성하는 단계;
(f) 상기 양자 포함 모드에서, Layer A, 미리 정해진 구획자, 및 상기 트레일러를 순차적으로 결합하여 단일 페이로드 문자열을 출력하는 단계;
(g) 상기 양자 부재 모드에서, 상기 구획자 및 상기 트레일러를 생성하지 아니하고 Layer A 만으로 단일 페이로드 문자열을 구성하여 출력하는 단계;
를 포함하며,
상기 단일 페이로드 문자열은, Layer A 가 상기 구획자 이전에 배치되어 상기 트레일러의 디코딩 없이도 분리 가능하고, 상기 이진 헤더의 Layer B 길이 필드 및 Layer C 길이 필드가 상기 트레일러의 디코딩 후 Layer B 및 Layer C 의 바이트 경계를 결정 가능하게 하는 wire format 구조를 갖는 것을 특징으로 하는 방법.

**[청구항 2] 내장 서명을 이용한 오프라인 검증 방법**

단일 페이로드 문자열을 입력받아 계층 데이터를 분리하고, 복합 모드인 경우 디지털 서명을 검증하는 방법으로서, 상기 단일 페이로드 문자열은 표준 QR 코드 데이터로 인코딩 가능하며 표준 QR 리더의 판독 결과 문자열의 선두에 평문으로 위치하는 공개 텍스트 계층인 Layer A, 미리 정해진 구획자, 및 트레일러로 해석되는 후속 문자열 영역을 순차적으로 결합한 복합 모드 구조의 페이로드이거나, 또는 상기 구획자를 포함하지 아니하고 Layer A 만으로 구성되는 단순 모드 구조의 페이로드이며, 상기 복합 모드 구조에서 상기 트레일러는, 후속 단계 (b) 에서 텍스트 안전 인코딩 방식에 대응한 디코딩 및 이진 헤더 파싱의 대상이 되는 문자열 영역으로서, 상기 디코딩 및 파싱이 정상적으로 수행되는 경우 버전 정보, 컨텍스트 메타데이터 계층 (Layer B) 의 바이트 길이 필드, 및 디지털 서명 계층 (Layer C) 의 바이트 길이 필드를 포함하는 이진 헤더, Layer B, 및 Layer C 를 순차적으로 결합한 프레임으로 디코딩되는 문자열 영역이며, 본 검증 방법은,
(a0) 입력 페이로드 문자열 내 상기 구획자의 존재 여부를 판별하는 단계;
(a1) 상기 구획자가 존재하지 아니하는 단순 모드인 경우, 입력 페이로드 문자열 전체를 Layer A 로 반환하고 후속 단계 (a) 내지 (c) 를 수행하지 아니하는 단계;
(a2) 상기 구획자가 존재하는 복합 모드인 경우에 후속 단계 (a) 내지 (c) 를 수행하는 단계;
(a) 입력 페이로드 문자열 내에서 상기 구획자가 최초로 출현하는 위치를 기준으로, 해당 위치 이전의 부분을 Layer A 로, 해당 위치 이후의 부분을 상기 트레일러로 분리하는 단계;
(b) 상기 트레일러를 상기 텍스트 안전 인코딩에 대응하여 디코딩한 후, 상기 이진 헤더를 파싱하여 후속 계층 데이터를 추출하되, 상기 후속 계층 데이터는 컨텍스트 메타데이터 계층(Layer B) 및 디지털 서명 계층(Layer C) 양자를 포함하며, Layer B 는 영(零) 바이트 길이를 가지는 경우를 포함하는 단계;
(c) 상기 버전 정보, Layer A 및 Layer B 를 포함하여 결정론적으로 직렬화된 서명 대상 메시지에 대하여, Layer C 를 디지털 서명으로 사용하여 검증 결과를 산출하는 단계;
를 포함하며,
상기 검증 대상 데이터 및 서명값은 동일 페이로드 문자열 내부에 포함되고, 상기 서명 검증 단계는 (i) 페이로드 문자열 내부에 포함된 상기 검증 대상 데이터 및 서명값과, (ii) 검증 절차 시작 이전에 이미 사전 저장되었거나 외부 입력 채널을 통해 검증기 메모리 또는 저장소에 가용한 상태인 공개키를 입력으로 사용하여 수행되며, 본 발명의 검증기 및 본 발명의 시스템서버 외부의 제3자 검증 인프라 — 제3자 온라인 검증 서버, 블록체인 검증 노드, 외부 인증 데이터베이스, 또는 제3자 인증 서비스 — 에 대한 호출은 수행하지 아니하고; 단, 본 발명의 시스템서버가 사용자 단말과 분리되어 분산 실시되는 경우 사용자 단말 ↔ 본 발명의 시스템서버 간의 페이로드 전달은 본 한정의 외부 호출에 해당하지 아니하는 것을 특징으로 하는 방법.

### 9.2 종속항 후보

**[청구항 3]** 청구항 1 에 있어서, 상기 이진 헤더는 1 바이트 버전, 2 바이트 Layer B 길이, 2 바이트 Layer C 길이를 big-endian 으로 직렬화한 5 바이트로 구성되고, 상기 텍스트 안전 인코딩은 base64 인코딩인 것을 특징으로 하는 방법.

**[청구항 4]** 청구항 1 에 있어서, 상기 구획자는 줄바꿈 문자로 둘러싸인 고유 문자열로서 Layer A 가 해당 문자열을 포함하는 입력은 거부되는 것을 특징으로 하는 방법.

**[청구항 5]** 청구항 2 에 있어서, 청구항 2 의 단계 (a0) 의 상기 구획자의 존재 여부 판별은 입력 페이로드 문자열 내에서 상기 구획자에 해당하는 고정 문자열 패턴을 검색하는 디코더 동작에 의해 수행되고, 단계 (a1) 의 단순 모드 처리는 입력 페이로드 문자열에 대한 트레일러 디코딩 동작 및 서명 검증 동작을 호출하지 아니하는 것을 특징으로 하는 방법.

**[청구항 6]** 청구항 2 에 있어서, 상기 디지털 서명은 Ed25519 서명 알고리즘에 따른 서명값이고, 청구항 2 의 단계 (c) 에서의 검증 결과 산출은 Ed25519 검증 알고리즘에 따라 수행되며, 상기 공개키는 디코더 측에 사전 저장된 신뢰 레지스트리 또는 외부 입력으로부터 선택되고, Layer C 의 바이트 길이가 Ed25519 서명 길이인 64 바이트와 일치하지 아니하는 경우 청구항 2 의 단계 (c) 에서의 검증 결과는 실패로 산출되는 것을 특징으로 하는 방법.

**[청구항 7]** 청구항 2 에 있어서, 디코더 또는 응용 프로그램에 반환되는 계층 데이터를 출력 정책에 따라 선택하는 리졸버를 더 포함하며, 상기 리졸버는 청구항 2 의 단계 (a2) 에 따라 단계 (a) 내지 (c) 가 수행되어 Layer B 가 추출된 복합 모드의 경우에,
(i) 제1 출력 수준에서 Layer A 만을 반환하고,
(ii) 제2 출력 수준에서 Layer A 및 Layer B 를 반환하되, Layer B 의 검증 상태를 미검증으로 표시하며,
(iii) 제3 출력 수준에서 상기 서명 검증이 성공한 경우에 한하여 Layer A 및 Layer B 를 검증된 데이터로 반환하고 검증 결과를 함께 출력하며,
청구항 2 의 단계 (a1) 에 따라 단순 모드로 처리된 경우에는, 어느 출력 수준이 요청된 경우에도 Layer A 만을 반환하는 것을 특징으로 하는 방법.

**[청구항 8]** 청구항 7 에 있어서, 상기 리졸버는,
(a) 청구항 2 의 단계 (a2) 에 따른 복합 모드 처리 중, 단계 (a) 에 따라 Layer A 와 트레일러가 분리된 이후에 단계 (b) 의 트레일러 디코딩 실패, 상기 이진 헤더의 버전 또는 길이 검증 실패, 또는 상기 이진 헤더의 길이 필드와 디코딩된 프레임의 실제 길이 간의 정합성 검증 실패 중 어느 하나가 발생하는 경우, 제2 출력 수준 또는 제3 출력 수준이 요청된 경우에도 Layer B, Layer C 및 Layer B 에 기초한 렌더링 결과를 반환하지 아니하고 상기 Layer A 및 실패 상태 정보를 포함하는 공개 수준 결과만을 반환하며,
(b) 공개키 부재 또는 상기 서명 검증 실패 중 어느 하나가 발생하는 경우, 제3 출력 수준이 요청된 때에 한하여 Layer B, Layer C 및 Layer B 에 기초한 렌더링 결과를 반환하지 아니하고 상기 Layer A 및 실패 상태 정보를 포함하는 공개 수준 결과만을 반환하도록 구성되는 것을 특징으로 하는 방법.

**[청구항 9]** 청구항 2 에 있어서, 페이로드 문자열 내 Layer A 의 선두 영역에는 발급자 식별자가 배치되어 있고, 본 검증 방법은 Layer A 의 선두 영역으로부터 상기 발급자 식별자를 추출하는 단계 및 복수의 검증 공개키 중 상기 발급자 식별자에 대응하는 공개키를 상기 서명 검증 단계의 입력으로서 선택하는 단계를 더 포함하는 것을 특징으로 하는 방법.

**[청구항 10]** 청구항 1 또는 청구항 2 의 방법을 컴퓨터에 실행시키기 위한 명령어가 기록된 컴퓨터 판독 가능 저장 매체.

**[청구항 11]** 청구항 1 의 방법을 수행하는 인코더, 청구항 2 의 방법을 수행하는 디코더 및 검증부, 그리고 청구항 7 의 출력 정책 및 청구항 8 의 안전 강등 정책을 수행하는 리졸버를 포함하는 단일 QR 페이로드 계층형 데이터 처리 시스템.

**[청구항 12]** 청구항 2 에 있어서, 상기 서명 대상 메시지는 (i) 미리 정해진 매직 식별자, (ii) 1 바이트 버전 필드, (iii) Layer A 의 UTF-8 바이트 길이를 나타내는 길이 필드, (iv) Layer A 의 UTF-8 바이트열, (v) Layer B 의 바이트 길이를 나타내는 길이 필드, (vi) Layer B 의 바이트열을 결정론적인 순서로 결합한 정규화된 바이트열로 구성되고, 상기 길이 필드들은 빅엔디안 부호없는 정수로 직렬화되는 것을 특징으로 하는 방법.

**[청구항 13]** 청구항 7 에 있어서, 상기 서명 대상 메시지가, 매직 식별자, 버전 필드, Layer A 의 UTF-8 바이트 길이를 나타내는 길이 필드, Layer A 의 UTF-8 바이트열, Layer B 의 바이트 길이를 나타내는 길이 필드, 및 Layer B 의 바이트열을 결정론적인 순서로 결합한 정규화된 바이트열로 구성되어 Layer B 의 바이트 길이가 상기 서명 대상에 결정론적으로 포함되며, 상기 제3 출력 수준에서 상기 서명 검증이 성공하고 Layer B 의 바이트 길이가 영(零) 인 경우, 상기 리졸버는 출력 데이터 구조에 Layer B 에 대응하는 필드를 존재 상태로 포함시키고, 상기 Layer B 에 대응하는 필드의 값을 영 바이트 길이의 빈 바이트열로 설정하며, 상기 Layer B 에 대응하는 필드와 연관된 검증 성공 상태 정보를 함께 출력함으로써, 상기 정규화 규칙에 따라 서명 검증된 영 바이트 길이의 Layer B 와, Layer B 가 페이로드에 존재하지 아니하는 상태가 출력 데이터 구조에서 서로 구별되도록 반환하는 것을 특징으로 하는 방법.

**[청구항 14]** 청구항 2 에 있어서, 상기 입력 페이로드 문자열은 휴대용 단말의 이미지 센서를 통해 촬영된 표준 QR 코드 심벌이 표준 QR 디코더에 의해 디코딩됨으로써 수신되고, 상기 휴대용 단말은, 복합 모드에서는 상기 표준 QR 디코더의 출력 문자열에서 상기 구획자 이전의 Layer A 를 보존하고, 단순 모드에서는 상기 출력 문자열 전체를 Layer A 로 보존한 상태로 청구항 7 의 출력 수준에 대응하는 반환 데이터를 시각, 청각, 및 촉각 출력 수단 중 적어도 하나를 통해 출력하되, 청구항 8 의 강등 정책이 발동된 경우에는 Layer B 및 Layer C 에 기초한 표시 객체를 생성하지 아니하고 Layer A 및 실패 상태 정보만을 출력 인터페이스에 전달하는 것을 특징으로 하는 방법.

**[청구항 15]** 청구항 11 의 단일 QR 페이로드 계층형 데이터 처리 시스템에 있어서, 상기 검증부는 매직 식별자, 버전 필드, Layer A 의 UTF-8 바이트 길이를 나타내는 길이 필드, Layer A 의 UTF-8 바이트열, Layer B 의 바이트 길이를 나타내는 길이 필드, 및 Layer B 의 바이트열을 결정론적인 순서로 결합한 정규화된 바이트열을 재구성하여 디지털 서명 검증을 수행하고, 상기 리졸버는 상기 서명 검증이 성공하고 Layer B 의 바이트 길이가 영(零) 인 경우 출력 데이터 구조에 Layer B 에 대응하는 필드를 존재 상태로 포함시키고 상기 Layer B 에 대응하는 필드의 값을 영 바이트 길이의 빈 바이트열로 설정하여 출력 인터페이스에 전달하도록 구성되는 것을 특징으로 하는 시스템.

## 11. 요약서 (Abstract)

본 발명은 표준 QR 코드의 데이터 필드에 인코딩 가능한 단일 페이로드 문자열 내에서, 표준 QR 리더에 의해 선두 평문으로 판독 가능한 공개 계층(Layer A)과, 미리 정해진 구획자 이후에 배치되는 텍스트 안전 인코딩된 트레일러를 결합하는 계층형 QR 페이로드 직렬화 및 검증 시스템을 제공한다. 상기 트레일러는 버전 정보 및 계층 길이 정보를 포함하는 이진 헤더와, 컨텍스트 메타데이터 계층 및 디지털 서명 계층을 포함할 수 있다. 디코더는 페이로드에 포함된 검증 대상 데이터 및 서명값과, 사전에 보유하거나 외부에서 제공된 공개키를 사용하여 온라인 검증 서버 호출 없이 서명 검증을 수행한다. 또한 리졸버는 실패 유형에 따라 두 그룹으로 구분되는 안전 강등 정책을 적용하되, 구조 파싱 실패 그룹(구획자 최초 출현 위치 기준 Layer A·트레일러 분리 이후의 트레일러 디코딩·이진 헤더 검증·길이 정합성 검증 실패)에서는 제2 및 제3 출력 수준 양자 모두 Layer A 만을 반환하고, 검증 실패 그룹(공개키 부재·서명 검증 실패)에서는 제3 출력 수준에 한하여 Layer A 만을 반환한다. 본 발명은 QR 모듈 또는 ECC 코드워드를 은닉 채널로 변조하지 않으며, 표준 QR 리더와 부분 호환되는 wire-format 결합과 검증 결과 기반 계층형 출력 정책을 동시에 제공한다. 또한 본 발명의 일 실시예는 검증부가 미리 정해진 매직 식별자, 버전, 계층 길이 필드 및 계층 바이트열을 결정론적인 순서로 결합한 정규화된 서명 대상 메시지에 대하여 서명 검증을 수행하고, 서명 검증이 성공하고 컨텍스트 메타데이터 계층의 길이가 영(零)인 경우에도 해당 빈 바이트열을 검증된 데이터로서 부재(null)로 변환하지 아니하고 보존하여 반환하며, 휴대용 단말의 이미지 센서를 통해 페이로드 문자열을 수신하여 검증 결과 및 출력 수준에 대응하는 반환 데이터를 시각적·청각적·촉각적으로 출력한다.
